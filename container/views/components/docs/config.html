Configs

App level configurations is placed under “./config” path, this includes:

  • Container
  • Authentication
  • Cors
  • Database
  • Server
  • Storage

Container

[./config/container.js] this configuration holds the settings for the app, including chasi instance.

| container.modelsDir:[Array]

the instance will autoload schemas in the listed dirs inside this property
default settings:

/**
  * Chasi will autoload this dirs,
  * look for models / schemas
  * then bind it to
  * [Event|Controller]Class
  */
  modelsDir:  [
  'container/Models/'
  ],

| container.blueprint:[Object]

this is where you can register your kernel file to control the View module and operations
default settings:

/**
  * View Blueprints
  * kernel file locations
  */
  blueprint:  {
     config:  'default',  //default view kernel to use
     stack:  {
       'default':  '/container/views/kernel.js',
     }
  },

| container.LoadPack:[Object]

Registry for Chasi's service extension packages
default settings:

/**
  * install out of the box packages
  * into the app container
  */
  LoadPack:  {
     'Sessions':  'package/framework/storage/session::init',
     'FileManager':  'container/modules/FileManager::init',
  },

| container.ServiceBootstrap:[Object]

Service Registry,the registered services will be available in chasi's
  instance, and will be accessible in the container, controllers and events.
  this config is also where you can implement microservices arch.

default settings:

ServiceBootstrap:  {
     'routers':  'container/services/RouterServiceProvider',
     'auth':  'container/services/AuthServiceProvider',
     'sockets':  'container/services/SocketServiceProvider',
  }

| container.middlewares:[Object]

any middlewares pointed to a route or route group 
  or even in a route containers, must be registered here.
  this config is autoloaded on boot.

default settings:

middlewares:  {
     /**
     * Register your middlewares here.
     * [alias] => require("<your middleware path in (container/middlewares/*)>")
     * after registration the middlewares will then
     * be installed to that Chasi Route MWs Repository,
     * then you can use it in your routes
     */
  }

Server

[./config/server.js] this configuration sets the server environment.

| server.enableSocketServer: [bool]

the instance will automatically boot the
  socketServer if enabled

| server.enableSocketServer: [Array]

A list of middlewares that will be automatically implemented
  on all socket connections.

| server.port:[Number]

a [BreakExecutionError] will be thrown if the port selected
  is unavailable.

| server.environment: [String]

the configuration which the handler will use for starting the server

| server.mode: [Object]

a way to setup a server environment, so switching environment will 
  be easier, if you have multiple environments.
  *Note: it is recommended to remove extra parameters on [live]* 
  
mode:  {
    dev:  {
      key:  checkout(process.env.SSLcontainerKey,  'local'),
      cert:  checkout(process.env.SSLcontainerCrt,  'local'),
      protocol:  'https',
    },
    local:  {
      key:  checkout(process.env.SSLcontainerKey2,  'null'),
      cert:  checkout(process.env.SSLcontainerCrt2,  'null'),
      protocol:  'http',
    }
  },

Athentication

[./config/authentication.js] by default, Chasi provide an authentication[Auth0] module, but you can always create your own.

| authentication.gateway: [Object]

this will contain all RouteContainer configs, Containers
   must registered in this list, before it can be used.

the object must be named after the “RouteContainer.name”
listed in ‘./container/services/RouterServiceProvider.js’

chasi:  {
    driver:  'jwt',
    enabled:  false,
    sessions:  true,
    key:  'Chasi',
    model:  'users',
    AuthRouteExceptions:  [
      /**
      * when enabled, all routes will
      * Be protected by [JWT] when
      * to excempt some routes,
      * it should be registered here..
      */
    ]
  }
